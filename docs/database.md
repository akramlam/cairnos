# Database

CairnOS stores all of its data in a single local **SQLite** file accessed through **[Drizzle ORM](https://orm.drizzle.team/)** (`drizzle-orm/better-sqlite3`). The `@cairn/server` engine owns the file via `better-sqlite3`, and both the UI (over HTTP) and the MCP server (via `@cairn/core`) read and write the same database. WAL mode is what makes that concurrent, multi-process access safe.

## At a glance

| Aspect | Value |
| --- | --- |
| Engine | SQLite (`better-sqlite3`) |
| ORM | Drizzle (`drizzle-orm/better-sqlite3`) |
| Schema | `packages/db/src/schema.ts` |
| Migrations | `packages/db/migrations` (generated by `drizzle-kit`) |
| Path resolver | `packages/db/src/paths.ts` |
| Default file | `%APPDATA%\Cairn\cairn.db` (Windows) |
| Concurrency | WAL mode + `busy_timeout` |
| Tables | 11 |

## Client configuration

When the `better-sqlite3` connection is opened, the following pragmas are applied:

```sql
PRAGMA journal_mode = WAL;     -- concurrent readers/writers across processes
PRAGMA foreign_keys = ON;      -- enforce FK constraints (SET NULL / CASCADE)
PRAGMA busy_timeout = <ms>;    -- wait instead of failing on a locked DB
```

WAL (Write-Ahead Logging) is essential: the Tauri webview cannot open the native SQLite file directly, the MCP server runs as a separate Node process, and both must share `cairn.db`. WAL allows these processes to read and write concurrently without corrupting the file.

## Conventions

- **IDs** - every primary-key `id` is a **UUID v4** stored as `TEXT`.
- **Timestamps** - every `*_at` column is an **ISO-8601** string stored as `TEXT` (e.g. `2026-06-20T14:32:00.000Z`). The same string flows unchanged through the DB, API, and UI, so there is zero serialization mapping.

## Migrations

Migrations are SQL files under `packages/db/migrations`, generated from the schema by `drizzle-kit`.

| Command | Purpose |
| --- | --- |
| `pnpm db:generate` | Diff `schema.ts` and emit a new migration SQL file |
| `pnpm db:migrate` | Apply pending migrations to the database |
| `pnpm db:seed` | Populate the database with seed data |

The database **self-migrates on boot**: both the `@cairn/server` engine and the `@cairn/mcp-server` apply any pending migrations automatically when they start, so the schema is always up to date wherever the DB is first opened.

## Tables

### `projects`

The top-level container that tasks, ideas, notes, and reminders can belong to.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | TEXT | Primary key - UUID v4 |
| `name` | TEXT | Project name |
| `description` | TEXT | Nullable |
| `objective` | TEXT | Nullable |
| `status` | TEXT | Default `active` - one of `active` \| `paused` \| `completed` \| `archived` |
| `priority` | TEXT | Default `medium` - one of `low` \| `medium` \| `high` |
| `color` | TEXT | Default `#3B82F6` |
| `created_at` | TEXT | ISO-8601 |
| `updated_at` | TEXT | ISO-8601 |

**Indexes:** `status`.

### `item_tags`

Polymorphic join table linking a tag to any taggable item. A single row attaches one tag to one item of a given type.

| Column | Type | Notes |
| --- | --- | --- |
| `tag_id` | TEXT | FK → `tags.id`, **ON DELETE CASCADE** |
| `item_type` | TEXT | One of `task` \| `project` \| `idea` \| `note` |
| `item_id` | TEXT | The id of the tagged item |

**Primary key:** composite `(tag_id, item_type, item_id)`.
**Indexes:** `(item_type, item_id)`.

Because the relationship is polymorphic, the same `tags` row can be reused across tasks, projects, ideas, and notes without a dedicated join table per type.

```
                         ┌──────────────┐
                         │   projects   │
                         └──────┬───────┘
                                │ (project_id, SET NULL on all children)
        ┌───────────────┬───────┼───────────────┬───────────────┐
        │               │       │               │               │
   ┌────▼────┐     ┌────▼───┐  ┌▼──────┐    ┌────▼────┐          │
   │  tasks  │     │ ideas  │  │ notes │    │reminders│          │
   └────┬────┘     └────────┘  └───────┘    └────▲────┘          │
        │                                        │               │
        └────────────── task_id (SET NULL) ──────┘               │
                                                                 │
   ┌──────────┐        ┌────────────┐                            │
   │   tags   │◄──CASCADE──│ item_tags │── (item_type, item_id) ─┘
   └──────────┘        └────────────┘   polymorphic → task/project/idea/note

   ┌─────────────┐  CASCADE  ┌────────────────────────┐
   │ brain_dumps │◄──────────│ classification_results │
   └─────────────┘           └────────────────────────┘

   ┌───────────────┐   ┌────────────┐
   │ activity_logs │   │  settings  │   (standalone)
   └───────────────┘   └────────────┘
```

### Referential integrity

- **Deleting a project never orphans its children.** Every `project_id` (and the `task_id` on `reminders`) foreign key uses **ON DELETE SET NULL**, so a deleted project simply detaches its tasks, ideas, notes, and reminders rather than removing them.
- **Tags cascade.** Deleting a tag removes its `item_tags` rows (**ON DELETE CASCADE**), and deleting a brain dump removes its `classification_results` rows (**ON DELETE CASCADE**).
